
## Folder Organization Rules

1. **Root-Level Folders**
   - Create folders for each major section of your documentation
   - Match folder names with your navigation structure
   - Example:
     ```
     src/content/
     ├── introduction/
     ├── getting-started/
     ├── guides/
     └── api/
     ```

2. **File Naming**
   - Use kebab-case for file names (`my-page.mdx` instead of `myPage.mdx`)
   - Always use `.mdx` extension
   - Make file names descriptive but concise
   ```
   getting-started/
   ├── quick-start.mdx
   ├── installation.mdx
   └── basic-configuration.mdx
   ```

3. **Nested Folders**
   - Create subfolders for related content
   - Match nested folders with your navigation groups
   - Keep nesting to a maximum of 2-3 levels
   ```
   api/
   ├── authentication/
   │   ├── oauth.mdx
   │   └── jwt.mdx
   └── endpoints/
       ├── users.mdx
       └── products.mdx
   ```


## Content Organization Best Practices

1. **Logical Grouping**
   - Group related content in the same folder
   - Use clear, descriptive folder names
   - Match folder structure with navigation
   ```
   guides/
   ├── basics/
   │   ├── markdown.mdx
   │   └── formatting.mdx
   └── advanced/
       ├── components.mdx
       └── customization.mdx
   ```

2. **Assets Organization**
   - Create an `assets` folder within each content section
   - Use subfolders for different types of assets
   ```
   getting-started/
   ├── installation.mdx
   └── assets/
       ├── images/
       │   └── setup-screenshot.png
       └── diagrams/
           └── architecture.svg
   ```


## Mapping Content to Navigation

Your folder structure should mirror your `site.config.json` navigation:

```json
{
  "navigation": [
    {
      "group": "Getting Started",
      "pages": [
        "getting-started/quickstart",
        "getting-started/installation",
        {
          "group": "Configuration",
          "pages": [
            "getting-started/configuration/basic-setup",
            "getting-started/configuration/advanced-options"
          ]
        }
      ]
    }
  ]
}
```


## Basic Template Structure

Every MDX file should follow this basic structure:

```mdx
---
title: Your Page Title
description: A brief description of the page content (recommended: 150-160 characters)
lastUpdated: YYYY-MM-DD
category: Category Name
---

# Main Title

Content goes here...
```

## Frontmatter

The frontmatter section is enclosed between `---` markers and contains metadata about the page.

### Required Fields

```yaml
---
title: Page Title          # Appears in navigation and as the main heading
description: Brief description of the content
lastUpdated: 2024-03-20    # Date in YYYY-MM-DD format
category: Category Name    # Used for grouping and filtering
---


## Common Patterns

1. **Index Files**
   - Use `index.mdx` for section landing pages
   ```
   api/
   ├── index.mdx          # API Overview
   └── authentication/
       ├── index.mdx      # Authentication Overview
       ├── oauth.mdx
       └── jwt.mdx
   ```

2. **Shared Resources**
   - Create a `_shared` folder for reusable content
   ```
   src/content/
   ├── _shared/
   │   ├── components/
   │   └── snippets/
   └── docs/
   ```

3. **Media Organization**
   - Keep media files close to their content
   ```
   getting-started/
   ├── installation.mdx
   ├── images/
   │   └── install-steps.png
   └── videos/
       └── tutorial.mp4
   ```

## Tips for Maintenance

1. **Regular Cleanup**
   - Remove unused files and folders
   - Keep asset folders organized
   - Update or archive outdated content

2. **Consistent Naming**
   - Follow the same naming convention throughout
   - Use descriptive but concise names
   - Document any special naming requirements

3. **Documentation Updates**
   - Keep navigation and content in sync
   - Update related files when moving content
   - Maintain a changelog for major structure changes
