## Nested Group

Key points about nested groups:

1. **Structure**
   - Nested groups are defined as objects within the `pages` array
   - Each nested group has its own `group` name and `pages` array
   - You can nest groups up to two levels deep (recommended maximum for good UX)

2. **File Organization**
   - Files should be organized in corresponding nested folders
   - Example path: `src/content/core-concepts/components/buttons.mdx`

3. **URL Structure**
   - URLs will reflect the nested structure
   - Example: `/docs/core-concepts/components/buttons`

4. **Best Practices**
   - Use nested groups sparingly to maintain clear navigation
   - Keep the hierarchy shallow (maximum 2-3 levels)
   - Group related content logically
   - Use clear, descriptive names for groups

### Complete Example with Nested Groups

```json
{
  "primaryTab": {
    "name": "Documentation"
  },
  "tabs": [
    {
      "name": "API Reference",
      "url": "api"
    }
  ],
  "navigation": [
    {
      "group": "",
      "pages": ["introduction/introduction"]
    },
    {
      "group": "Getting Started",
      "pages": [
        "getting-started/quickstart",
        {
          "group": "Configuration",
          "pages": [
            "getting-started/config/basic-setup",
            "getting-started/config/advanced-options"
          ]
        },
        {
          "group": "Customization",
          "pages": [
            "getting-started/customization/themes",
            "getting-started/customization/layouts"
          ]
        }
      ]
    }
  ]
}
```


## Tab-Specific Groups

When organizing content for different tabs (like "Documentation", "API", "Advanced"), you can create tab-specific groups in the navigation. Here's how to structure it:

```json
{
  "primaryTab": {
    "name": "Documentation"
  },
  "tabs": [
    {
      "name": "API Reference",
      "url": "api"
    },
    {
      "name": "Advanced",
      "url": "advanced"
    }
  ],
  "navigation": [
    // Documentation Tab Groups
    {
      "group": "Getting Started",
      "pages": [
        "getting-started/quickstart",
        "getting-started/installation"
      ]
    },
    // API Reference Tab Groups
    {
      "group": "API Reference",
      "pages": [
        "api/introduction",
        {
          "group": "Endpoints",
          "pages": [
            "api/endpoints/authentication",
            "api/endpoints/users",
            "api/endpoints/products"
          ]
        },
        {
          "group": "Error Handling",
          "pages": [
            "api/errors/status-codes",
            "api/errors/error-responses"
          ]
        }
      ]
    },
    // Advanced Tab Groups
    {
      "group": "Advanced",
      "pages": [
        "advanced/overview",
        {
          "group": "Performance",
          "pages": [
            "advanced/performance/optimization",
            "advanced/performance/caching"
          ]
        },
        {
          "group": "Security",
          "pages": [
            "advanced/security/authentication",
            "advanced/security/authorization"
          ]
        }
      ]
    }
  ]
}
```

### How Tab-Specific Groups Work

1. **Tab Definition**
   - First, define your tabs in the `tabs` array
   - Each tab needs a `name` and `url`
   ```json
   {
     "tabs": [
       {
         "name": "API Reference",
         "url": "api"
       }
     ]
   }
   ```

2. **Group Association**
   - Groups are associated with tabs by matching the group name with the tab name
   - For example, groups with "API Reference" will appear in the API tab
   ```json
   {
     "group": "API Reference",
     "pages": [
       "api/introduction",
       "api/authentication"
     ]
   }
   ```

3. **URL Structure**
   - URLs should match the tab's url prefix
   - Example: For API tab (`url: "api"`), use paths like:
     - `api/introduction`
     - `api/endpoints/users`

### Tab-Specific Groups

1. **Naming Convention**
   - Use consistent naming between tabs and their corresponding groups
   - Keep group names clear and descriptive

2. **Content Organization**
   - Organize content files in directories matching the tab structure
   - Example file structure:
   ```
   src/content/
   ├── introduction/
   ├── getting-started/
   ├── api/
   │   ├── rest/
   │   └── graphql/
   └── advanced/
       └── deployment/
   ```

3. **Navigation Flow**
   - Keep related content within the same tab
   - Use clear hierarchies within each tab
   - Limit nesting to 2-3 levels for better user experience

4. **URL Consistency**
   - Maintain consistent URL patterns within each tab
   - Use the tab's URL prefix for all content within that tab
   - Example:
     - `/api/rest/authentication`
     - `/advanced/deployment/docker`
